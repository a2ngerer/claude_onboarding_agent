# End-User Hooks Broad Rollout — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Introduce a shared hook-emission helper and integrate it into the four setup skills where a strong hook recipe exists (`devops-setup`, `web-development-setup`, `data-science-setup`, `academic-writing-setup`). Establish the `_plugin` / `_skill` marker convention so plugin-owned hook entries can be identified for refresh and removal.

**Architecture:** Markdown refactor of existing SKILL.md files plus one new shared helper (`skills/_shared/emit-hook.md`) and four generated hook scripts emitted into the user's project at runtime. No executable code in this repo beyond the scripts that generator skills emit — SKILL.md files remain prompts Claude follows.

**Tech Stack:** Markdown, Claude Code skill framework, bash + `jq` (the scripts generated by skills use the same toolchain as the existing `.claude/hooks/check-dependencies.sh`). Verification is grep-based (static) plus a documented manual E2E walkthrough.

**Spec:** `docs/superpowers/specs/2026-04-21-end-user-hooks-rollout-design.md` — read it first. Every task below references decisions made there.

- [ ] **Step 1: Verify the target path is currently empty**

- [ ] **Step 2: Create `skills/_shared/emit-hook.md` with the following content**

```markdown
# Emit Hook Protocol

This file is read by every setup skill that writes end-user Claude Code hooks into `./.claude/settings.json`. It defines the merge-and-emit procedure so each skill stays focused on its own hook catalog and does not reimplement the file-handling logic.

Current consumers: `devops-setup`, `web-development-setup`, `data-science-setup`, `academic-writing-setup`. The shared plugin-internal hook at `.claude/hooks/check-dependencies.sh` in this repo is a development tool and is NOT emitted by this helper — this helper is for user-facing setup skills only.

## Inputs (set by the calling skill before reading this file)

- `skill_slug` — the directory name of the calling skill (e.g. `devops-setup`). Written into the `_skill` marker.
- `hook_entries` — a list of hook specifications to write. Each entry is an object with:
  - `event` — one of `PreToolUse`, `PostToolUse`, `SessionStart` (others allowed if Claude Code supports them; the helper is event-agnostic).
  - `matcher` — the matcher string (e.g. `Bash`, `Edit|Write`, `startup|resume`). May be omitted for events that do not use matchers.
  - `command` — the shell command to execute. For non-trivial logic, this should be `bash "$CLAUDE_PROJECT_DIR/.claude/hooks/<name>.sh"` and the calling skill must also emit the script file.
  - `script_source` — (optional) if present, the helper writes this string to `./.claude/hooks/<script_name>` with mode 0755 before touching `settings.json`. The calling skill is responsible for generating the script content; the helper handles file placement and executable bits.
  - `script_name` — (required when `script_source` is present) the filename under `./.claude/hooks/`.

Every hook entry the helper writes carries these two fields at the top of the inner `hooks[]` object:

- `"_plugin": "claude-onboarding-agent"`
- `"_skill": "<skill_slug>"`

## Step H1 — Verify `.claude/` exists

Create `./.claude/` if it does not exist (`mkdir -p .claude`). Do not touch anything inside it yet.

## Step H2 — Read existing settings.json

If `./.claude/settings.json` does not exist: set `settings: {}` in memory and skip to Step H4.

Otherwise read the file and attempt a JSON parse.

- Parse success: keep the parsed object as `settings`.
- Parse failure: print (adapt to detected language):

  > "⚠ `.claude/settings.json` is not valid JSON. I will not touch it. Here are the hook entries you can paste in manually once the file parses:"

  Then print the raw hook JSON the helper would have written. Set `settings_write_skipped: true` and stop the helper. The calling skill continues with its remaining steps (hooks are best-effort, not a blocker).

## Step H3 — Remove prior plugin entries for this skill

For each top-level hook event key in `settings.hooks` (if present):

1. For each matcher-group in the event's array:
   - Filter the inner `hooks[]` array: drop every entry whose `_plugin == "claude-onboarding-agent"` AND `_skill == <skill_slug>`.
   - If the resulting `hooks[]` array is empty, drop the whole matcher-group.
2. If the event's array is now empty, drop the event key.
3. If `settings.hooks` is now empty (`{}`), drop the `hooks` key.

This step is what makes the procedure idempotent: a re-run removes the skill's previous entries before writing fresh ones.

## Step H4 — Emit script files

For each `hook_entries[i]` where `script_source` is set:

1. Resolve the target path: `./.claude/hooks/<script_name>`.
2. Write `script_source` to the file, overwriting if it exists.
3. Set mode 0755 (`chmod +x`).

Every generated script MUST start with this comment block so identification survives hand-edits to settings.json:

```
#!/usr/bin/env bash
# Generated by claude-onboarding-agent (skill: <skill_slug>)
# Purpose: <one-line purpose>
# Safe to delete — Claude Code will continue without the hook.
```

## Step H5 — Append fresh plugin entries

Ensure `settings.hooks` exists (`settings.hooks = settings.hooks || {}`).

For each `hook_entries[i]`:

1. Ensure `settings.hooks[event]` exists as an array.
2. Find a matcher-group with `matcher == <matcher>` in that array. If none exists, append a new group `{ "matcher": "<matcher>", "hooks": [] }`. If the event does not use matchers (e.g. `SessionStart`), find or append a group with no matcher field.
3. Append the new hook entry to the matcher-group's `hooks[]` array. The entry MUST include `_plugin`, `_skill`, `type: "command"`, and `command` — in that order, with `_plugin` and `_skill` first so the markers are visible when the file is scanned by eye.

## Step H6 — Write settings.json

Serialize `settings` back to `./.claude/settings.json`:

- Two-space indent.
- Trailing newline.
- Key order within existing objects preserved where the JSON library allows.

Never write the file if Step H2 set `settings_write_skipped: true`.

## Step H7 — Report status to the calling skill

Set these variables for the calling skill's completion summary:

- `hooks_emitted_count` — number of entries actually written.
- `hooks_script_files` — list of script paths written under `./.claude/hooks/` (empty if no `script_source` was provided).
- `hooks_write_skipped` — true if Step H2 bailed out due to corrupt settings.json.

The calling skill uses these to print a line like:

```
Hooks:
  [✓ 1 entry written to .claude/settings.json + .claude/hooks/<name>.sh
   | ⚠ settings.json is corrupt — hook entries printed above for manual paste
   | — skipped per user]
```

## Failure modes (documented so consumers know what to expect)

- Corrupt `settings.json` — the helper does NOT overwrite. It prints the fallback and returns.
- Hook command whose matcher is already in use by a user-authored entry — the helper appends into the existing matcher-group. The user entry is never modified.
- Re-run after the user manually deleted a plugin script file — the helper re-emits the script. Settings.json entry is idempotent via Step H3 (remove-then-append).
- Re-run after the user manually removed the markers from a plugin entry — the helper treats it as user-authored and does not remove it. A second entry gets written. This is a user-editing foot-gun, not a bug; the markers are documented as the ownership signal.
```

On `yes`:

   - Extracts `.tool_input.command` via `jq`.
   - If the command string matches `terraform apply`, `pulumi up`, `kubectl apply`, `aws cloudformation deploy`, or `az deployment ... create`, emits a JSON object with `hookSpecificOutput.hookEventName = "PreToolUse"` and `hookSpecificOutput.additionalContext` reminding Claude to run the corresponding plan/preview first.

   # Generated by claude-onboarding-agent (skill: devops-setup)
   # Purpose: remind Claude to plan/preview before an IaC apply.
   # Safe to delete — Claude Code will continue without the hook.

   set -u

   INPUT=$(cat)
   CMD=$(printf '%s' "$INPUT" | jq -r '.tool_input.command // empty' 2>/dev/null)
   [ -z "$CMD" ] && exit 0

   emit() {
     jq -cn --arg ctx "$1" \
       '{hookSpecificOutput: {hookEventName: "PreToolUse", additionalContext: $ctx}}'
   }

   case "$CMD" in
     *"terraform apply"*)
       emit "About to run \`terraform apply\`. Confirm you have run \`terraform plan\` against the same config and reviewed the diff. If not, run \`terraform plan\` first and paste the plan summary before applying."

     *"pulumi up"*)
       emit "About to run \`pulumi up\`. Confirm you have run \`pulumi preview\` and reviewed the diff. If not, run \`pulumi preview\` first."
       ;;
     *"kubectl apply"*)
       emit "About to run \`kubectl apply\`. Confirm the manifest matches the intended change — consider \`kubectl diff -f <manifest>\` first, especially on production clusters."
       ;;
     *"aws cloudformation deploy"*)
       emit "About to run \`aws cloudformation deploy\`. Confirm a change-set review has happened — consider \`aws cloudformation deploy --no-execute-changeset\` first."
       ;;
     *"az deployment"*"create"*)
       emit "About to run \`az deployment create\`. Consider \`az deployment <scope> what-if\` first to preview the change."
       ;;

   exit 0
   ```
